# AI 辅助根因分析（RCA）复盘 · 工作流模板

## 适用场景

- 线上事故 / 故障复盘
- 部署演练 / 灰度发布后的问题复盘
- 测试发现的成组缺陷复盘（不是单 bug，是同源问题）
- 任何需要回答"问题之后还有没有同源风险"的场景

## 核心方法论：4 要素 + 5-Why + 3 层追问

```
输入：现象描述 + 时间线 + 影响范围 + 已知线索（4 要素）
   ↓
工具：5-Why 纵向深挖（直到出现"架构/流程/认知"层面的回答）
   ↓
扩展：AI 3 层追问（检测维度 / 流程维度 / 同类风险）
   ↓
输出：单点根因 + 同类风险清单 + 改进措施（带责任方和时间）
```

## 四步流程

### Step 1：准备四要素材料（5min）

任何复盘开始前必须备齐：

| 要素 | 关键问题 | 常见素材 |
|------|---------|---------|
| 现象描述 | 什么"不对"了？用户/系统的可见表现是什么？ | 错误日志、监控截图、用户反馈 |
| 时间线 | 什么时候开始？什么时候发现？什么时候定位/修复？ | 监控时间戳、发布记录、Jira 流转 |
| 影响范围 | 哪些用户/服务/数据受影响？严重度多少？ | 流量数据、订单量、错误率 |
| 已知线索 | 已经查到的代码位置、日志关键字、配置项 | 排查记录、初步猜测 |

> ⚠️ 缺少任何一项都会让 5-Why 走偏。AI 没有上下文时只会瞎编。

### Step 2：5-Why 纵向深挖（10min）

给 AI 的 Prompt 模板：

```
请基于以下信息，对问题 [问题描述] 做 5-Why 根因分析。

【现象】{现象描述}
【时间线】{时间线}
【影响范围】{影响范围}
【已知线索】{代码/日志/配置线索}

要求:
1. 每一层 Why 都基于上一层的"果",不要跳跃
2. 至少深挖 5 层
3. 当回答触及"架构假设/流程缺陷/团队认知"层面时,标记为【根因】
4. 不要止步于"这是个 bug",要回答"为什么会出现这个 bug"
```

**判断 5-Why 走到位的信号**：

- ✅ 最后一层回答的是"假设/规范/认知"层面，不是"代码/配置"层面
- ✅ 修复方案不再是"改这一行"，而是"改进流程/补充规范"
- ❌ 如果最后一层还在说"因为没考虑到 XXX"，那是借口，不是根因

### Step 3：AI 3 层追问（10min）

5-Why 只能挖深度，挖不到广度。补充三个追问：

| 维度 | 追问句式 | 价值 |
|------|---------|------|
| **检测维度** | "为什么这个缺陷没在 UT/IT/CodeReview/灰度阶段被拦截？" | 暴露质量保障缺口 |
| **流程维度** | "团队的什么规范/流程/工具如果存在，就能避免这个问题？" | 转化为可执行的改进措施 |
| **同类风险** | "代码库里还有哪些组件可能存在同源的问题？给我一份排查清单" | 防止同源问题复发 |

> 💡 同类风险追问是 RCA 复盘和事后报告的核心差异。前者会主动外推。

### Step 4：输出结构化报告（10min）

报告必备 8 个章节：

```
1. 复盘目的（要回答的核心问题，不是流水账）
2. 事件时间线（带"观察"的时间线，不只是时刻表）
3. 5-Why 深度分析（每个问题或问题组一段）
4. 多层根因归纳（表层/中层/深层）
5. 拦截环节复盘（应有拦截手段 vs 实际状态）
6. 同类风险外推（4 种以上风险类型）
7. 改进措施（按优先级 + 责任方 + 截止 + 验收标准）
8. 经验沉淀（带走 3 件事）
```

## Prompt 模板：一键生成完整复盘

```
你是一名资深的质量分析专家，请基于以下材料生成一份完整的 RCA 复盘报告。

【背景】
{项目背景、为什么要做这个复盘}

【问题清单】
{逐个列出问题，包括现象/时间线/影响/已知线索/当前状态}

【输出要求】
1. 包含以下章节：复盘目的 / 时间线 / 5-Why 分析 / 多层根因 / 拦截复盘 / 同类风险 / 改进措施 / 经验沉淀
2. 5-Why 必须挖到"架构假设/流程缺陷/团队认知"层面才停
3. 改进措施必须带优先级（P0/P1/P2）、负责方、截止时间、验收标准
4. 同类风险至少列 4 类，每类说明排查范围和严重度
5. 经验沉淀部分给 3 条"带得走"的洞察，不要重复前面的内容

格式要求：Markdown，含表格，便于团队评审使用。
```

## 常见陷阱

| 陷阱 | 表现 | 避免方法 |
|------|------|---------|
| **5-Why 走形** | 5 层都在说"代码层面"，没触及架构/流程 | 强制要求最后一层回答"假设/规范/认知" |
| **根因泛化** | 根因写成"测试不够""人员不足"等虚词 | 要求每个根因可对应至少 1 条改进措施 |
| **只修不防** | 改进措施只针对当前问题，没有外推 | 必须包含"同类风险排查"措施 |
| **责任甩锅** | 把根因归结为"某人没注意" | 系统层面的问题不应归咎个人，要问"为什么这个流程允许个人疏漏导致事故" |
| **AI 瞎编上下文** | AI 把不存在的代码/配置说得很真 | Step 1 必须提供真实素材，关键结论必须人工核对 |

## 效率对比

| 环节 | 纯手工 | AI 辅助 | AI 擅长点 |
|------|--------|---------|----------|
| 5-Why 推演 | 40min（容易走形或停太早） | 10min | 不知疲倦地追问，不会"差不多就行" |
| 同类风险外推 | 30min（往往遗漏） | 5min | 系统性扫描代码库可能的同源点 |
| 报告结构化 | 60min | 10min | 套用模板填充 |
| 措辞润色 | 30min | 5min | 团队/管理层双受众适配 |
| 人工核对 | — | 15min | 数据准确性 + 根因合理性 |
| **总计** | **~3h** | **~45min** | |

## 适用范围扩展

这套方法不止于线上故障复盘：

- **测试缺陷复盘**：对一个迭代的所有 bug 做根因聚类分析
- **演练复盘**：BCP/容灾/混沌工程演练后的问题归纳
- **个人项目复盘**：把"踩坑记录"升级为"踩坑根因 + 同类风险预防"
