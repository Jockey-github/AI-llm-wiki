# AI Agent 架构全景 — LLM / Tools / MCP / Skill 的关系

> 沉淀时间: 2026-04-30 | 来源: 实际构建 tes_expense / daily-debate / jira-mcp 等项目的经验总结

## 一句话总结

**LLM 是大脑（想），Tools 是手脚（做），MCP 是工具的标准接口（连），Skill 是经验手册（知）。Agent 就是把这四者串成一个自主决策循环。**

## 核心工作原理

传统 LLM 调用是"一问一答"。Agent 的本质区别是引入了**自主决策循环**：

```
用户指令
   ↓
┌─────────────────────────────┐
│  LLM 推理：理解任务，决定下一步  │ ← 大脑
│         ↓                    │
│  选择并调用 Tool              │ ← 手脚
│         ↓                    │
│  获取 Tool 执行结果            │
│         ↓                    │
│  LLM 再次推理：够了吗？        │
│    ├── 不够 → 继续调用 Tool    │ ← 循环
│    └── 够了 → 输出最终结果     │
└─────────────────────────────┘
```

关键点：**LLM 自己决定调什么工具、调几次、什么时候停**。

## 四个核心组件

### 1. LLM — 大脑

负责**推理、规划、决策**。它不直接执行任何操作，但它决定：
- 任务该怎么拆解
- 下一步该调什么工具
- 工具返回的结果是否满足需求
- 什么时候该停下来给用户答复

### 2. Tools — 手脚

LLM 本身只能"想"和"说"，不能"做"。Tools 是执行器，分两类：

| 类型 | 说明 | 实际例子 |
|------|------|---------|
| **本地工具** | 在本机执行的能力 | Bash（跑命令）、Read/Write（读写文件）、Glob/Grep（搜索代码） |
| **远程工具** | 调用外部服务的能力 | Jira MCP（查缺陷）、Confluence MCP（读文档）、TES MCP（审批） |

每个工具有一个 JSON Schema 描述名称、参数、用途。LLM 看到描述后就知道什么场景该调什么工具。

### 3. MCP Server — 标准化的工具提供者

MCP（Model Context Protocol）是一个**标准协议**，解决的问题是：让 LLM 以统一方式接入各种外部系统。

类比：**工具的 USB 接口标准**。

```
没有 MCP 之前：
  LLM → 自定义代码 → Jira API
  LLM → 另一套自定义代码 → Confluence API
  LLM → 又一套代码 → TES API
  （每个都要单独适配，格式不统一）

有了 MCP 之后：
  LLM → MCP 协议 → Jira MCP Server → Jira API
  LLM → MCP 协议 → Confluence MCP Server → Confluence API
  LLM → MCP 协议 → TES MCP Server → TES API
  （统一协议，即插即用）
```

### 4. Skill — 打包好的领域知识

Skill 不是工具，而是**知识包**。它告诉 LLM：
- 在特定场景下该怎么做（流程、步骤）
- 有哪些坑要避开（经验沉淀）
- 该用什么工具、怎么用

**Skill 是"知道怎么做"，Tool 是"能做到"**。两者缺一不可。

以 tes_expense skill 为例：

```
Skill 提供的是知识：
  - TES 的 DOM 结构长什么样（slide-frame 体系）
  - 保存按钮要限制 x > 800
  - 发票金额 > 60 要截断
  - 加班类型选择器要用 .searcher-list-item 精确匹配

Tool 提供的是能力：
  - Bash 执行 Playwright 脚本
  - Read 读取脚本文件
  - Write 修改脚本
```

## 实际执行流程示例

### 示例 1：查 Jira 缺陷

用户说："使用 jira-mcp 帮我看一下最近一周内的线上问题单"

1. **LLM 推理**：用户要查 Jira 缺陷，需要用 jira-mcp 的 search_issues 工具
2. **调用工具**：`mcp__jira-mcp__search_issues(jql="...", fields="...")`
3. **获取结果**：拿到 JSON 格式的缺陷列表
4. **LLM 再推理**：用户要表格展示，把 JSON 格式化成 Markdown 表格
5. **输出结果**：展示给用户

### 示例 2：TES 报销自动化

用户说："使用报销 skill 帮我上传费用"

1. **加载 Skill**：读取 tes_expense 的 SKILL.md，获取完整操作知识
2. **LLM 规划**：需要执行 Playwright 脚本，参数是日期列表
3. **调用 Bash 工具**：执行 `python tes_create_expense.py --date 0421,0422,0423`
4. **观察输出**：脚本执行日志
5. **LLM 判断**：是否全部成功？有没有报错？
6. **反馈给用户**：汇报执行结果

## 延伸：Agent 的三种实现模式

| 模式 | 说明 | 适用场景 |
|------|------|---------|
| **单次调用** | 一问一答，无循环 | 分类、摘要、提取 |
| **工作流（Workflow）** | 代码控制调用顺序，LLM 不自主决策 | 固定流程的多步任务 |
| **自主 Agent** | LLM 自己决定调什么、调几次 | 开放式、探索性任务 |

Claude Code 本身就是一个自主 Agent —— 你给它一个任务，它自己决定读哪些文件、改哪些代码、跑什么命令、什么时候停。
