# AI 测试用例生成 Prompt 模式

从实战中提炼的 Prompt 模板，覆盖测试用例生成、代码审查、Bug 调试三个高频场景。

---

## 1. Zero-shot 直接生成（快速版）

适用：需求明确、场景简单、赶时间

```markdown
你是一名资深测试工程师。根据以下需求，生成测试用例表格：

【需求】{粘贴需求描述}

要求：
- 格式：编号 | 场景 | 前置条件 | 步骤 | 预期结果 | 优先级
- 覆盖：正向(50%) + 反向(30%) + 边界(20%)
- 优先级：P0(核心功能) / P1(重要场景) / P2(边缘场景)
- 每个用例独立可执行
```

## 2. Few-shot 带示例生成（质量版）

适用：复杂业务、需要特定风格、首次用 AI 生成

```markdown
你是一名资深测试工程师。参考以下示例风格，为【新需求】生成测试用例。

【示例用例】
| 编号 | 场景 | 前置条件 | 步骤 | 预期结果 | 优先级 |
|------|------|---------|------|---------|--------|
| TC-01 | 正常下单-微信支付 | 用户已登录，购物车有商品 | 1.进入结算页 2.选微信支付 3.确认支付 | 支付成功，订单状态=已支付 | P0 |
| TC-02 | 下单-库存不足 | 商品库存=0 | 1.进入商品页 2.点击立即购买 | 提示"库存不足"，无法下单 | P1 |

【新需求】{粘贴需求}
按上述格式生成，覆盖正向/反向/边界。
```

**实测效果**：Few-shot 比 Zero-shot 的用例格式一致性提升 40%，边界覆盖多 30%。

## 3. 多轮拆解法（复杂需求）

适用：需求文档超过 3 页、涉及多个子系统、新人接手的模块

```markdown
# 第一轮：理清范围
你是一名测试架构师。阅读以下需求文档，帮我梳理：
1. 涉及的功能模块有哪些？
2. 各模块之间的依赖关系？
3. 哪些是核心路径、哪些是分支路径？
4. 有哪些隐含场景（文档没说但可能存在）？

【需求文档】{粘贴}

# 第二轮：设计测试框架
基于上一轮的模块分解，为每个模块设计测试策略：
- 需要哪些类型的测试（接口/UI/数据验证/性能）
- 测试数据的准备策略
- 模块间的集成测试点

# 第三轮：填充详细用例
基于前两轮的框架，逐个模块填充详细测试用例（格式同上）。
```

## 4. Bug 调试 Prompt 模板

从 Kafka 竞态调试案例中提炼的三要素法：

```markdown
我遇到一个问题：

【现象】{精确描述你看到了什么：日志/报错/行为}
【上下文】{环境：版本/配置/时间点；触发条件：做了什么操作}
【假设】{你怀疑可能是哪里的问题？}

请帮我：
1. 分析可能的根因（按可能性排序）
2. 建议验证方法（能证明或排除每个假设）
3. 如需要更多信息，告诉我具体看什么
```

**要点**：现象要精确（截图/日志片段）→ 上下文要完整（版本/时间/操作）→ 假设要诚实（不确定就说"不确定"）。

---

## 使用建议

| 场景 | 推荐模式 | 预计耗时 |
|------|---------|---------|
| 简单需求 | Zero-shot | 5 分钟 |
| 正规项目/新人上手 | Few-shot | 10 分钟 |
| 复杂需求/大型系统 | 多轮拆解 | 30 分钟 |
| 疑难 Bug | 三要素法 | 取决于问题 |
